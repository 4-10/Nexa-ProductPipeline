# Comparative Analysis

## Purpose

Comparative analysis helps Product Pipeline avoid designing in a vacuum.

The comparison should include direct competitors when they exist, adjacent workflow tools, manual workarounds, and the base-shell alternative of doing nothing yet.

## Comparison Dimensions

- Target user and buyer.
- Core promise.
- Key workflows.
- Pricing or business model when relevant.
- UX posture and emotional tone.
- Switching cost.
- Data or integration moat.
- Weaknesses and frustration points.
- Opportunities for Nexa to differentiate.

## Output Shape

Use a compact structure:

```text
Alternative:
Who it serves:
What it does well:
Where it falls short:
What Product Pipeline should learn from it:
Implication for first wedge:
```

## Differentiator Rules

A differentiator should be more than "AI-powered" or "better UX." It should connect to a concrete user pain, workflow change, economic advantage, emotional relief, or speed-to-outcome advantage.

## Emotional Calling

Capture the product's emotional center in plain language:

- What frustration does this relieve?
- What confidence does it create?
- What identity or aspiration does it support?
- What would make the user feel "finally, this gets it"?

This does not replace functional requirements. It sharpens them.
