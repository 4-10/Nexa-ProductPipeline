# Intake To Artifacts

## Purpose

This file defines the early artifact ladder Product Pipeline should guide an agent through when the human gives a high-level idea.

## Intake Style

The intake should feel collaborative and senior, not like a rigid form. The agent should ask one useful question at a time when the idea is still blurry, then become decisive when enough context exists.

The goal is to capture:

- product idea,
- target user or buyer,
- user pain,
- desired outcome,
- constraints,
- first version appetite,
- examples or competitors,
- emotional tone,
- existing parent KB facts.

## Artifact Ladder

```text
Idea
-> intake notes
-> research brief
-> comparative analysis
-> opportunity brief
-> epic map
-> story/task slices
-> ready-for-dev issues
```

## Artifact Definitions

### Intake Notes

The raw but structured capture of human guidance. This should preserve wording that reveals intent or emotional tone.

### Research Brief

Evidence and unknowns gathered before the backlog is produced.

### Comparative Analysis

A concise market or workflow comparison against alternatives, competitors, and analogs.

### Opportunity Brief

The product direction: target user, problem, desired outcome, differentiator, first wedge, risks, and recommended next move.

### Epic Map

A small set of capability areas. Product Pipeline may stop at epic level if story/task slicing would be premature.

### Story/Task Slices

Bounded work that implementation agents can build, verify, and close.

## Rule Of Thumb

If the product direction is uncertain, create a research spike or epic. If the behavior is clear and testable, create a story or task.
