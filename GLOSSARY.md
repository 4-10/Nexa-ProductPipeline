# Nexa Product Pipeline Glossary

## Core Terms

- **NoderaOS** - current working name for the broader agentic operating system vision.
- **Nexa-AOS** - repo and working system name for the Nexa core operating layer.
- **Product Pipeline** - module that turns human product intent into researched, structured, developable work.
- **Parent KB** - the project knowledge base that the final Product Pipeline module should read from and update. The final module should not own a separate runtime KB.
- **Working KB** - this repository's temporary build-time knowledge base for designing and implementing the module.
- **AppGenerator / App Builder** - companion module path that creates the base app shell Product Pipeline will help focus, brand, and turn into a product.
- **Base AppBuilder shell** - a domain-neutral, unbranded, unfocused starter application that can become a specific product after product guidance.

## Product Artifacts

- **Human guidance** - initial product idea, target direction, constraints, appetite, and strategic intent provided by the human.
- **Intake** - the structured first conversation that captures the idea without overfitting too early.
- **Research brief** - evidence-gathering artifact covering market, alternatives, customers, constraints, and unknowns.
- **Comparative analysis** - structured comparison against alternatives, competitors, analogs, and incumbent workflows.
- **Target market focus** - the selected audience, buyer, user, use case, and first wedge.
- **Differentiator thesis** - why this product should win or matter compared with alternatives.
- **Emotional calling** - the human-level pull of the product: the feeling, frustration, relief, aspiration, or identity that makes the work more than a feature list.
- **Opportunity brief** - concise synthesis of customer, pain, market, differentiator, risks, and recommended direction.
- **Epic** - a coherent product capability area large enough to contain multiple developable tasks.
- **Story** - user- or system-centered behavior slice that can be tested and shipped.
- **Task** - bounded implementation or research work item with acceptance criteria and verification.
- **Ready-for-dev gate** - the checkpoint that decides whether a work item is clear enough for a build agent to claim.

## Operating Terms

- **GitHub Project** - live planning board for this module's work: `https://github.com/users/4-10/projects/4`.
- **Audit log** - append-only account of meaningful changes to this KB or module plan.
- **Decision log** - rationale for non-obvious choices that future agents should not relitigate casually.
- **Subagent briefing** - pre-flight routing doc for agents that start without full conversation context.

When you find yourself explaining a Product Pipeline term in a session, add it here and register the term in `INDEX.md` if it becomes a routing keyword.
